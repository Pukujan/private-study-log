## The Corrected Approach

Cortex should NOT follow:
> lean MVP → prove one feature → expand toward the final system

Cortex SHOULD follow:
> freeze the complete final service contract → map every existing mechanism into that contract → eliminate competing authorities → wire all promised layers into one route → validate the complete route independently → release only when the contract is complete

### Contract-complete, not feature-incomplete

The first releasable Cortex may support a bounded set of domains, but it cannot omit any layer that defines Cortex.

The first release MUST include:
1. Runtime service contract
2. Proactive context intelligence
3. Iterative research and narrowing
4. Human outcome and scope lock
5. Strong/weak model routing
6. Governed execution
7. Independent verification
8. Bounded rework
9. Automatic closeout
10. Outcome and downstream-cost measurement
11. Operational MAPE-K observation and proposal loop

MAPE-K may return INSUFFICIENT_HISTORY rather than changing anything. That is acceptable. The loop exists and behaves honestly; it is not postponed as a future subsystem.

Proactive injection may return NO_RELEVANT_CONTEXT. Confidence forecasting may return UNKNOWN. An oracle may return ABSTAIN. Those are complete behaviors, not missing features.

## What Must Be Frozen Before Implementation

### 1. One complete player contract

| Player | Goal | Frozen Output |
|---|---|---|
| Human | Define intent and acceptable trade-offs | Outcome/scope approval or rejection |
| Runtime adapter | Connect any host to Cortex | Normalized, correlated tool and lifecycle events |
| Enforcement hook | Prevent ungoverned actions from retaining governed status | Permit, block, pause or downgrade |
| Cortex kernel | Own run identity, state, scope, transition legality | Authoritative transition event |
| Context intelligence | Surface relevant prior knowledge and conflicts | Bounded context packet or explicit abstention |
| Research engine | Resolve decision-relevant uncertainty iteratively | Evidence bundle and sufficiency status |
| Strong driver | Narrow, plan, decompose, recommend | Evidence-linked outcome proposal |
| Weak workers | Perform bounded labor | Artifacts and execution receipts |
| Model router | Match roles to qualified available models | Bound route receipt or UNRESOLVED |
| Independent oracle | Determine contract satisfaction | PASS, FAIL, ABSTAIN, UNRESOLVED or environment failure |
| Scribe/projector | Make the run understandable | Generated closeout and human result view |
| Learning optimizer | Compare predictions and outcomes | Bounded optimization proposal or no-change verdict |

No player may silently absorb another player's authority.

### 2. One canonical run contract

Every governed task must carry one immutable identity chain:
```
request_id → decision_id → outcome_contract_id → run_id → route_id → worker_task_ids → artifact_ids → oracle_receipt_ids → closeout_id → outcome_measurement_ids
```

No separate ledger should independently invent another identity for the same decision.

### 3. One authoritative event model

Context packets, confidence records, risk tradeoffs, closeouts and optimization observations should NOT become five new canonical stores. They should be typed events connected to the same run:

intent_recorded, context_surfaced, research_claim_recorded, uncertainty_recorded, tradeoff_recorded, forecast_recorded, outcome_locked, route_bound, artifact_produced, verification_completed, human_result_recorded, downstream_effect_observed, optimization_proposed

Human-readable packets, reports and dashboards are projections.

### 4. A frozen conformance suite

| Damage | Required Proof |
|---|---|
| Wrong interpretation | No mutation before confirmed outcome |
| Rebuilt existing work | Relevant prior design surfaced before planning |
| No research | No outcome lock without sufficient evidence or explicit abstention |
| Weak plan | Acceptance checks bound before execution |
| Scope expansion | Out-of-scope native tool action blocked or contract formally revised |
| Self-certification | Builder output cannot mint final success |
| Lost decision | Restart restores the exact contract and state |
| Repeated failure | Matching KEDB pattern surfaced and control applied |
| Docs/state/code disagreement | Generated views reconcile to events and artifacts |
| Deferred maintenance cost | Later regressions and rework tied back to the decision |

These tests must be frozen before the implementation agent sees the hidden cases.

## Independent Validation, Not Cortex Grading Itself

The final validation should NOT be a single easy app_build trial. It needs at least:
- One complex software build
- One evidence-heavy research and decision task
- One maintenance or operational task involving existing project state
- More than one model family
- More than one worker capability tier
- Hermes plus a second real runtime before claiming runtime independence
- Hidden failure injections
- Restart and interrupted-run recovery
- Follow-up measurement after delivery, not only immediate test success

The builder may produce artifacts. Cortex may capture evidence. But the only success authorities should be frozen deterministic checks, independently controlled evaluation, and the human's outcome judgment where machine verification is impossible.

## Revised Verdict

The correct conclusion is NOT:
> Build the smallest useful Cortex first, then expand.

It IS:
> Freeze the complete Cortex contract first, consolidate existing mechanisms into it, and do not call any intermediate state Cortex v1.

Fable's four builds should not become four sequential product milestones. Their useful content must be absorbed into one frozen end-state:

narrowing + injection + research sufficiency + outcome lock + runtime enforcement + model routing + execution + independent verification + closeout + downstream measurement + constrained optimization

Then the work is not "adding Cortex features." It is tightening and wiring the layers that already exist until this single contract works end to end.

So yes: my earlier "vertical slice first" recommendation was still too close to the quick-win philosophy. The safer direction is a contract-complete first release with no deferred promised layers and no self-validation of the overall product claim.
